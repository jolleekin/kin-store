const todoSlice = createSlice({
  name: "todos",
  initialState: { items: [], status: "idle" } as TodoState,
  reducers: {
    toggle(state, action: PayloadAction<number>) {
      const todo = state.items.find(t => t.id === action.payload);
      if (todo) todo.done = !todo.done;
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchTodos.pending,   s => { s.status = "loading"; })
      .addCase(fetchTodos.fulfilled, (s, a) => { s.items = a.payload; })
      .addCase(fetchTodos.rejected,  s => { s.status = "failed"; })
  },
});
export const { toggle } = todoSlice.actions;

What about Zustand?

const useStore = create(
  devtools(
    persist(
      immer((set) => ({
        todos: [] as string[],
        addTodo: (todo: string) => set((state) => {
          state.todos.push(todo);
        }),
      })),
      { name: 'todo-storage' }

    { name: 'TodoStore' }

function TodoApp() {

  const addTodo = useStore(state => state.addTodo);

Problem 01

No type safety

state inside the immer callback is any. state in state.todos and state.addTodo is unknown — TypeScript can't help you here.

Problem 02

State and actions are indistinguishable

todos (data) and addTodo (a command) live in the same object with no structural distinction. You can't tell from the type alone what's state and what's behavior.

Problem 03

Stable refs forced through subscriptions

addTodo is stable — it will never change. Yet useStore(state => state.addTodo) registers a subscription that runs on every state update, forever.

Problem 04

Inside-out middleware pipeline

devtools(persist(immer(...))) reads left-to-right but executes right-to-left. You must mentally invert the nesting to understand the pipeline order.

Problem 05

Unpredictable API shape

Each middleware wraps the store itself, and can even alter its API. Two middlewares can make conflicting assumptions — what set does depends on composition order.

What about Jotai? 

import { atom, useAtomValue, useSetAtom } from 'jotai';

const todosAtom  = atom<Todo[]>([]);
const statusAtom = atom<'idle' | 'loading' | 'failed'>('idle');

// App logic must be wrapped in an atom — no plain function allowed.
const addTodoAtom = atom(null, (get, set, text: string) => {
  set(todosAtom, (prev) => [
    ...prev, { id: Date.now(), text, done: false },
  ]);
});
const fetchTodosAtom = atom(null, async (get, set) => {
  set(statusAtom, 'loading');
  try {
    const todos = await fetch('/api/todos').then(r => r.json()) as Todo[];
    set(todosAtom, todos); set(statusAtom, 'idle');
  } catch { set(statusAtom, 'failed'); }
});
// Must use hooks to call write atoms — can't call from outside React.
function TodoApp() {
  const todos  = useAtomValue(todosAtom);
  const status = useAtomValue(statusAtom);
  const addTodo    = useSetAtom(addTodoAtom);
  const fetchTodos = useSetAtom(fetchTodosAtom);}
Problem 01

Actions must be atoms

Even a simple addTodo must be wrapped in atom(null, (get, set, arg) => ...). Every piece of logic is a write atom — there is no plain function style.

Problem 02

Logic can't run outside React

Write atoms are activated via useSetAtom. You can't call them from a service layer, a test, or a timeout without reaching for getDefaultStore().

Problem 03

Hard to debug at scale

Stack traces surface at the useSetAtom call site in your component — not at the atom definition. A chain of atoms triggering other atoms is hard to trace in a debugger.

What about MobX? 

class TodoStore {
  todos: Todo[] = [];
  status: 'idle' | 'loading' | 'failed' = 'idle';

  constructor() {
    makeAutoObservable(this);  }

  addTodo(text: string) {
    this.todos.push({ id: Date.now(), text, done: false });
  }

  async fetchTodos() {
    this.status = 'loading';
    try {
      const todos = await fetch('/api/todos').then(r => r.json()) as Todo[];
      runInAction(() => {
        this.todos = todos;
        this.status = 'idle';
      });    } catch {            runInAction(() => { this.status = 'failed'; });    }
  }
}

const todoStore = new TodoStore();

const TodoApp = observer(() => {
  const { todos, status } = todoStore;
  return <button onClick={() => todoStore.addTodo('Buy groceries')}>Add</button>;
});
Problem 01

makeAutoObservable magic

Instruments every field and method invisibly — fields become observables, getters become computeds, methods become actions. No explicit list of what is reactive vs. not.

Problem 02

runInAction required in async

Mutations after an await must be wrapped in runInAction(). Forgetting causes silent stale-data bugs — no error is thrown, the UI just shows wrong data.

Problem 03

observer() on every component

Every component reading observable state must be wrapped in observer(). Forgetting also causes silent stale-data bugs — still no error thrown. Two silent failure modes.

Part 2 — Kin Store

Three primitives.

Each composable. None mandatory.

createStore
The irreducible floor. getState · setState · subscribe. Nothing else.
withPlugins
Opt-in structure: methods, reducers, middleware, lifecycle hooks, namespaced plugins.
derive
Lazy, dependency-tracked, read-only views composed from one or more stores.
Part 2 — Kin Store — First primitive

createStore

Step 1 — A value. 3 methods. Nothing else. 

type Todo = string;
type TodoState = { todos: string[]; status: "idle" | "loading" | "failed" };

const todoStore = createStore({ todos: [], status: "idle" } as TodoState);function addTodo(text: string): void {
  todoStore.setState((state) => ({ ...state, todos: [...state.todos, text] }));
}

async function fetchTodos(): Promise<void> {
  todoStore.setState((state) => ({ ...state, status: "loading" }));
  try {
    const resp = await fetch("/api/todos");
    const todos = (await resp.json()) as Todo[];
    todoStore.setState({ todos, status: "idle" });
  } catch {
    todoStore.setState((state) => ({ ...state, status: "failed" }));
  }
}addTodo('Hello world');todoStore.getState(); // { todos: ["Hello world"], status: "idle" }todoStore.subscribe((getState, prevState) => {
  console.log(prevState, "-->", getState());
});

createStore

Holds any value. Returns three methods: getState · setState · subscribe. Nothing else.

setState

Accepts a new value or an updater (state) => nextState. Notifies all subscribers.

Calling your functions

Logic lives in plain functions. No dispatch, no action types.

getState

Reads the current state synchronously. Always the latest value.

subscribe

Fires on every change. Receives a getState getter and the previous state.

withPlugins

Step 2 — Move logic inside the store. Still plain setState.

        try {

          const todos = (await resp.json()) as Todo[];
          store.setState({ todos, status: "idle" });
        } catch {
          store.setState((state) => ({ ...state, status: "failed" }));
        }
      },
    }),
  });todoStore.addTodo("Hello world");todoStore.getState();

  console.log(prevState, "-->", getState());
});

withPlugins

Upgrades/creates a store with plugin support. Each .use() adds capability — no nesting, no wrapping.

methods

Factory (store) => ({...}) that returns your logic as methods. Methods have full access to the store API.

Calling methods

Methods become properties of the store. No dispatch, no action types.

getState · subscribe

Identical to createStore — getState reads synchronously, subscribe fires on every change.

immer

Write reducers as draft mutations. No spread syntax — Immer handles immutability.

persist

Saves to localStorage on every change. Hydrates on startup. Custom storage, partial slices.

history

history.undo() · history.redo() · history.reset(). Works with any state change.

Part 2 - Kin Store - Third primitive

derive — reactive composition

Step 5 — Read from any store. The derived store tracks its sources. 

const userStore = createStore({ name: "Ada", role: "admin" });
const cartStore = createStore({ items: [] as string[], total: 0 });// Reads from both. Recomputes only when either changes.
const summary = derive((get) => ({
  greeting: `Hello, ${get(userStore).name}`,
  itemCount: get(cartStore).items.length,
  total: get(cartStore).total,
}));// Conditional — only subscribes to adminStore when role === "admin".
const view = derive((get) =>
  get(userStore).role === "admin"
    ? get(adminStore).dashboard
    : get(publicStore).feed
);view.destroy(); // Unsubscribes from all sources.
✦Auto-tracked—get() registers the dependency. No selector arrays.
✦Lazy—cold when no subscribers. Zero cost until something listens.
✦Conditional—branches only subscribe to stores they actually read.
✦Clean up when done—destroy() unsubscribes from all sources. No leaks.
No new paradigm. Just stores that talk to each other.
